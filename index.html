Highlight #1 Version Control Systems (VCS)

Version Control, also referred to as 'revision control' or 'source control', is basically a system that allows a user to track and retrieve their files over time. A Version Control System (VCS) allows users to track all file changes, who they were changed by, when they were changed and utilize specific versions of their projects at any given point in time. Version control is a concept that can be/is used by virtually anyone.

Think about it like this: have you ever updated a project or document and saved it as "Document2017" rather than saving it to the previous file named "Document2016?" VCS solves the problem (on a larger scale) that the "Save As" function somewhat-fixes. Yeah, it's that simple!

Highlight #2 What's So Great About Git?

There are several great qualities of Git that make it an ideal Version Control System to utilize with your projects and related works. I'll explain a few of my favorites.

Git is a distributed Version Control System, meaning that users have full, complete access to the server database from their remote connection. Being a distributed VCS is a huge advantage alone simply because if, for instance, multiple people were trying to work on the same project or if the server goes down, there is no need to worry!

Another advantage of Git is the fact that it has integrity. Because Git tracks every file you tell it to, there are virtually no changes to the files or directory that go unnoticed. One of the fundamential concepts behind Gits creation was (in my own words) 'No change left behind.'

Lastly, the best advantage (in my opinion) of using Git is that it is super easy to learn! It's so easy that by the end of Lesson 3, you'll be a pro at navigating the different commands used by Git: add, commit, status, pull, push, etc.

Highlight #3 Git Basics

Git is definitely better than the average when compared to your standard Version Control Systems. The major difference that sets Git apart from its competition is the fact that Git stores data fundamentally different than the average VCS. Git stores data as different versions of file(s) or "snapshots" as it is referred, compared to other VCS which store data as a set of changes that are made to an original set of files. Git's unique way of "thinking" allows for the retrieval of any version of any file, no matter if it's the first versio, the 17th version, or the latest version.

Highlight #4 Git's 3 Stages

There are 3 main stages of a file: modified, staged, and committed. These stages represent the continuous life cycle of the files in your Git project. A modified file is one that has been edited but not committed to your database. A staged file is one that has been modified and is set to be committed to a snapshot and saved to your repository. Lastly, a committed file is a file whose current version has successfully been modified and staged, and is now safely stored on your local database.

In addition to the 3 main stages of a file, there are 3 different areas that those files live in: the working directory, the staging area, or the Git directory. When a file is modified, it is in the working directory. And lastly, when a file is committed, a snapshot is taken in the staging area and the file is moved to the Git directory/local repository.

Highlight #5 Remote Repository vs Local Repository

A repository is, by definition, a location in which data is stored and managed. When talking about Git, there are 2 different types of repositories: remote and local. Allow me to explain what makes each type of repository unique.

(Hint: It's not a very difficult concept to grasp)

A remote repository is a repository that exists on a remote server. Access is shared among several team members.

A local repository is a repository that exists on a local machine of an individual user. There are 2 ways for a user to create a local repository on his/her local machine: from scratch or by cloning another, already existing remote repository.
